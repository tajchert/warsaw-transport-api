Community-written OpenAPI description of the ZTM/WTP transit endpoints of api.um.warszawa.pl

Unofficial, community-written OpenAPI description of the public-transport endpoints of the City of Warsaw open-data API (api.um.warszawa.pl).

Every operation here was verified live on 2026-06-28. Where the live behaviour differs from the official docs or community examples, this spec documents the live behaviour and flags the difference.

Things you must know before using this API

  • Errors are returned with HTTP 200. You must inspect the JSON body, not the status code. See the Result* schemas and the x-error-strings table below.
    • {"result": null} → query OK, no matching data
    • {"result": "Błędna metoda lub parametry wywołania"} → wrong action/id/params
    • {"result": "false", "error": "Błędny apikey lub jego brak"} → bad/missing key
    • {"result": "Wfs error: Connection reset"} → transient wfsstore_get error, retry
  • dbtimetable_get is a multiplexer: the id (dataset UUID) and which params you send select one of three different datasets. They are documented as three separate example sets on the one operation.
  • busestrams_get ignores line/brigade at present — filter client-side.
  • The vehicle feed contains stale "ghost" rows — drop entries whose Time is older than ~60–120 s.
  • No rate limit is published or enforced at normal volumes; an undocumented load-balancer abuse filter exists. Poll live data every ~15–20 s; cache static datasets daily. See the project guide (README) for the full polling policy.
  • Stop coordinates are not available from the API (legacy dbstore_get ids are decommissioned). Join the GTFS feed instead — see README.md §"GTFS join".

Authentication

Every transit call needs apikey as a query parameter. Get a free key by registering at https://api.um.warszawa.pl/.

Live positions

Real-time vehicle GPS.

Live vehicle positions (buses & trams)

Returns the current GPS positions of all buses (type=1) or all trams (type=2) across the city. No bounding-box filter — clip client-side.

Verified gotchas:

  • type is effectively required (1=bus → ~1000+ rows, 2=tram → ~200+ rows).
  • line and brigade are accepted but ignored — the full set is returned regardless. Filter on the Lines field yourself.
  • The feed keeps stale rows for vehicles that stopped reporting (timestamps can be hours/months old). Drop rows where Time is older than ~60–120 s.
  • Refreshes server-side every ~10–20 s; do not poll faster.

Timetables

Stop dictionaries and scheduled departures (dbtimetable_get).

Timetable datasets (multiplexed by `id`)

One action, three datasets selected by the id UUID plus the params you send. A stop is addressed by a pair: busstopId (4-digit zespół / stop-group) + busstopNr (2-char post, e.g. 01).

id dataset required params
88cd555f-6f31-43ca-9de4-66c479ad5942 Lines at a stop post busstopId, busstopNr
e923fa0e-d96c-43f9-ae6e-60518c9f3238 Scheduled departures (whole day) busstopId, busstopNr, line
b27f4c17-5c50-4a5b-89dd-236b282bc499 Stop group by name (coords) name

Result rows are key/value pair lists, not objects — fold them yourself (see schema). Scheduled times (czas) can exceed 24h (e.g. 25:30:00).

⚠️ The b27f4c17… "by name" dataset currently returns null for every query (server-side broken). Resolve names via public_transport_dictionary → zespoly_przystankowe, and get coordinates from the GTFS join (see README).

Reference data

Route topology and dictionaries; large, static-ish, cache daily.

Route definitions (line → variant → stop sequence)

Nested object: line → route-variant code → stop-sequence index → stop descriptor (nr_zespolu, nr_przystanku, ulica_id, typ, odleglosc). Resolve ulica_id and typ via public_transport_dictionary. Large & static-ish — cache daily. No dataset id; apikey only.

Dictionary (streets, stop-group names, stop types, places)

Returns four maps: ulice (street id→name), typy_przystankow (1=stały, 2=na żądanie, 3=krańcowy), zespoly_przystankowe (stop-group id→name — use this for name↔id resolution), and miejsca. No dataset id; apikey only. Cache daily.

Vector layers

WFS point layers (metro entrances, bikes, parking, …) with coordinates.

WFS point layers (metro entrances, bikes, parking, …)

Returns a point layer as featureMemberList with geometry.coordinates (latitude/longitude) plus key/value properties. The layer is chosen by id. Optionally constrain with bbox, circle, filter, limit.

Known layer ids (✅ = verified live):

id layer
0ac7f6d1-a26b-430f-9e3d-a41c5356b9a3 ✅ Metro/subway entrances
a08136ec-1037-4029-9aa5-b0d0ee0b9d88 ✅ Veturilo cycle stations
8a235d27-b96a-4876-9b92-9e164940c9b6 Cycle tracks
157648fd-a603-4861-af96-884a8e35b155 Parking lots
e26218cb-61ec-4ccb-81cc-fd19a6fee0f8 Theaters

There is no bus/tram stop layer and no tram-track layer here. May return "Wfs error: Connection reset" transiently — retry.
